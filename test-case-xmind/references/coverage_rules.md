# 覆盖规则

在阅读所有输入文档之后、编写 `cases.json` 之前，使用这些规则决定测试覆盖范围。

## 建立覆盖模型

识别以下内容：

- 用户角色和权限边界。
- 主流程和备选流程。
- 输入项、校验规则、默认值和边界值。
- UI 状态：初始态、加载态、成功态、空状态、禁用态、警告态、错误态。
- 后端契约：接口、请求字段、响应字段、状态码、持久化、幂等性、最终一致性。
- 需求、前端行为、后端行为和交互稿之间的冲突。

## 用例选择

优先生成能发现缺陷的用例：

- 每个核心流程至少覆盖一条正向路径。
- 每条有意义的校验规则至少覆盖一条反向用例。
- 对数字、日期、长度、文件大小、分页、配额和限流规则覆盖边界值。
- 权限用例需要覆盖可见 UI、隐藏 UI、直接调用接口、跨租户/跨用户数据访问。
- 失败用例需要覆盖网络错误、服务端错误、超时、重复提交、过期数据和部分成功。
- 多步骤流程需要覆盖状态流转。
- 只有当源文档提到平台、浏览器、响应式、语言、时区或无障碍要求时，才生成对应兼容性用例。

## 不同文档的信号

需求文档：

- 验收标准直接映射为核心用例。
- 业务规则通常需要同时生成正向和反向用例。

前端技术文档：

- 组件禁用态、前端校验、路由守卫、乐观更新和本地缓存需要独立用例。

后端技术文档：

- 必填字段、错误码、异步处理、重试、鉴权和数据一致性需要独立用例。

交互稿/原型：

- 空状态、加载态、文案、弹窗行为、确认流程、悬停/聚焦状态和导航分支，只要用户可见且影响体验，就需要覆盖。

## 质量标准

- 每条用例都应该让测试人员不重新阅读所有源文档也能执行。
- 尽量保持一个步骤一个动作。
- 预期结果必须可观察。
- 避免生成大量近似重复用例。
- 对推断出的行为记录假设。

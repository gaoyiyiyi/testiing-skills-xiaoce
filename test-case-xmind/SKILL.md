# Test Case XMind

## 概览

使用本 skill 将产品和技术资料转换成结构化测试用例，并导出 `.xmind` 文件。

用户的真实使用方式是：提供一个或多个输入文档，然后要求 Codex 使用本 skill 生成测试用例。不要要求用户手动运行脚本，除非用户正在调试或维护这个 skill。Codex 应该在生成结构化用例后自动调用脚本完成校验和导出。

用例必须包含：

- 用例标题
- 前置条件
- 步骤
- 每个步骤对应的预期结果

一条用例只有一个前置条件组，可以包含多个步骤。每个步骤必须且只能对应一个预期结果。

## 工作流程

1. 收集用户提供的源文档，检查所有会影响行为、数据、页面状态、权限、接口、校验、异常处理或边界场景的输入。
2. 在编写结构化用例 JSON 前，阅读 `references/case_schema.md`。
3. 在决定覆盖范围前，阅读 `references/coverage_rules.md`。
4. 根据 schema 生成结构化 `cases.json`。
5. 运行 `scripts/validate_cases.py` 校验 JSON，并修复所有错误。
6. 运行 `scripts/export_xmind.py` 生成 `.xmind` 文件。
7. 生成 `.xmind` 后结束操作。不要再生成 HTML 预览页，也不要尝试自动打开浏览器。

`.xmind` 文件是主要交付物。

## 输入文档分析

当用户提供多个源文档时，要交叉分析它们，而不是分别做摘要：

- 需求文档用于识别用户价值、业务规则和验收标准。
- 前端技术文档用于识别页面状态、路由、前端校验、浏览器约束和可见行为。
- 后端技术文档用于识别接口契约、持久化规则、权限、异步任务、幂等性和失败模式。
- 交互稿/原型用于识别页面流转、组件状态、文案、错误展示位置、空状态、加载态和禁用态。

如果文档之间存在冲突，优先在用例备注或假设中记录你的判断。只有当冲突会阻碍继续生成有价值的用例时，才向用户提问。

## 覆盖要求

生成可执行、可评审的测试用例。优先保证有意义的覆盖，而不是堆砌大量重复用例。

需要覆盖：

- 核心成功路径
- 必填项和校验失败
- 边界值
- 空状态、加载态和错误态
- 不同权限和角色差异
- 接口失败，以及可重试/不可重试错误
- 前后端数据一致性
- 如果源文档提到兼容性或响应式要求，也要覆盖相关场景

## 结构化输出

导出前先将用例保存为 JSON。最小示例：

```json
{
  "title": "登录功能测试用例",
  "modules": [
    {
      "name": "登录",
      "cases": [
        {
          "title": "使用有效账号和密码登录成功",
          "precondition": "已存在一个注册用户，且用户已打开登录页面。",
          "steps": [
            {
              "action": "输入有效账号和密码，并提交登录表单。",
              "expected": "用户认证成功，并跳转到默认落地页。"
            }
          ]
        }
      ]
    }
  ]
}
```

## 脚本

脚本是给 Codex 自动调用的稳定工具，不是用户的主要操作入口。用户只需要提供文档并提出生成请求。

在 skill 目录下调用：

```bash
python3 scripts/validate_cases.py cases.json
python3 scripts/export_xmind.py cases.json --output test-cases.xmind
```

这些脚本只使用 Python 标准库，避免安装额外依赖。

## 演示流程

准备丝滑演示时，阅读 `references/demo_flow.md`。推荐流程：

1. 根据输入文档生成并校验 `cases.json`。
2. 导出 `test-cases.xmind`。
3. 生成 `.xmind` 后结束操作，由用户自行在 XMind 中打开或导入。

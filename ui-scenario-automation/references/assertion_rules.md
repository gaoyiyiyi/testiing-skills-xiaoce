# UI Assertion Rules

Use assertions that reflect what a user can observe or what the browser can reliably verify.

## Good Assertions

- Text that confirms the outcome appears.
- The URL changed to the expected route.
- A button becomes enabled or disabled.
- A validation message appears near the form field.
- A modal opens or closes.
- A list count changes after filtering or creation.
- The selected option, checkbox state, or input value matches expectation.

## Avoid Weak Assertions

- Asserting only that a click succeeded.
- Asserting decorative text unrelated to the business outcome.
- Using brittle CSS classes generated by frameworks.
- Depending on animation timing without waiting for a stable state.
- Verifying implementation details that users cannot observe.

## Choosing UI vs API Automation

Prefer UI automation for:

- End-to-end user paths.
- Forms and validation messages.
- Navigation and routing.
- Permission-dependent visible controls.
- Modals, toasts, and user-facing status changes.

Prefer API automation for:

- Large parameter combinations.
- Permission matrices.
- Backend state transitions.
- Data setup and cleanup.
- Error contracts that do not require UI rendering.

## Locator Improvement Advice

When a target is hard to locate, recommend adding stable selectors in the product code:

```html
<button data-testid="submit-answer">提交答案</button>
```

Do not overuse `data-testid`. Prefer accessible roles and labels when they already exist because they improve both testability and accessibility.

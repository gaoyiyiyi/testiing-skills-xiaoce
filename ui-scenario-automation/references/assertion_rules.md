# UI 断言规则

UI 断言应尽量反映用户能看到的结果，或浏览器能稳定验证的状态。

## 推荐断言

- 页面出现能确认业务结果的文案。
- URL 跳转到预期路由。
- 按钮变为可点击或不可点击。
- 表单字段附近出现校验提示。
- 弹窗打开或关闭。
- 列表过滤、创建或删除后，数量发生预期变化。
- 选中项、复选框状态或输入框内容符合预期。

## 避免脆弱断言

- 只断言点击动作成功。
- 断言与业务结果无关的装饰性文案。
- 依赖前端框架生成的易变 CSS 类名。
- 依赖动画时间，而不等待稳定状态。
- 校验用户不可见的实现细节。

## 如何选择 UI 自动化或接口自动化

更适合 UI 自动化的场景：

- 端到端用户路径。
- 表单填写和校验提示。
- 页面导航和路由跳转。
- 权限影响下的可见按钮或入口。
- 弹窗、提示、用户可见状态变化。

更适合接口自动化的场景：

- 大量参数组合。
- 权限矩阵。
- 后端状态流转。
- 数据准备和清理。
- 不依赖页面渲染的错误契约。

## 定位器改进建议

如果页面元素难以稳定定位，可以建议在产品代码中增加稳定标识：

```html
<button data-testid="submit-answer">提交答案</button>
```

不要滥用 `data-testid`。如果页面已有良好的角色、标签和可访问名称，优先使用这些信息，因为它们同时有助于自动化稳定性和可访问性。
